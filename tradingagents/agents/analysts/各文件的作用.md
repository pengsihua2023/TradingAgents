这个 `tradingagents/agents/analysts` 文件夹里一共 4 个文件：

* `fundamentals_analyst.py`
* `market_analyst.py`
* `news_analyst.py`
* `social_media_analyst.py` ([GitHub][1])

它们的共同模式非常一致：**每个文件都定义一个 `create_xxx_analyst(llm)` 工厂函数，返回一个节点函数 `xxx_analyst_node(state)`，供上层 LangGraph 图注册为 analyst 节点使用。** 每个节点都会从 `state` 里读取 `trade_date` 和 `company_of_interest`，构造 `instrument_context`，绑定各自可用的工具，调用 `llm.bind_tools(tools)` 生成结果；如果这次结果不再包含 `tool_calls`，就把最终文本写入某个专属 report 字段。([GitHub][2])

你可以把这 4 个文件理解成：**4 种不同视角的前线分析员**。下面逐个说。

---

## 1. `fundamentals_analyst.py`

这个文件的作用是：**创建“基本面分析员”节点**。它定义了 `create_fundamentals_analyst(llm)`，返回内部函数 `fundamentals_analyst_node(state)`。这个节点从共享状态中读取当前交易日期 `trade_date`，并通过 `build_instrument_context(state["company_of_interest"])` 生成当前股票/公司的上下文。([GitHub][2])

### 它能调用哪些工具

它绑定了 4 个工具：

* `get_fundamentals`
* `get_balance_sheet`
* `get_cashflow`
* `get_income_statement` ([GitHub][2])

这说明它的职责很明确：不是看价格走势，也不是看新闻，而是专门看公司概况和财务报表。

### 它的提示词在要求什么

它的 `system_message` 明确要求模型：

* 分析公司过去一周内的基本面信息
* 尽可能详细写出公司画像、财务文档、财务历史
* 提供可执行、带证据支撑的洞见
* 在结尾附一个 Markdown 表格，整理关键信息。([GitHub][2])

### 它的输出写到哪里

执行后，如果这次 LLM 结果里已经没有工具调用，代码就把 `result.content` 存入：

* `fundamentals_report`

同时把本次消息对象放回：

* `messages: [result]`。([GitHub][2])

### 它在系统中的位置

它是 analyst 团队里负责“公司内在质量”的那位，重点回答的是：

* 公司财务健康吗
* 报表有什么红旗
* 盈利、现金流、资产负债情况怎样
* 从基本面看值不值得买。([GitHub][2])

---

## 2. `market_analyst.py`

这个文件的作用是：**创建“市场/技术分析员”节点**。同样，它定义了 `create_market_analyst(llm)`，返回 `market_analyst_node(state)`。这个节点也会从状态里取 `trade_date` 和 `company_of_interest`，并生成 `instrument_context`。([GitHub][3])

### 它能调用哪些工具

它绑定两个工具：

* `get_stock_data`
* `get_indicators` ([GitHub][3])

这说明它是四个 analyst 中最偏“技术分析/行情分析”的角色。

### 它的提示词有什么特点

这个文件的提示词是四个 analyst 里最长、最具体的一个。它不只是说“分析市场”，而是直接给了模型一整套候选指标清单，包括：

* 均线：`close_50_sma`、`close_200_sma`、`close_10_ema`
* MACD 系列：`macd`、`macds`、`macdh`
* 动量：`rsi`
* 波动率：`boll`、`boll_ub`、`boll_lb`、`atr`
* 成交量加权：`vwma`。([GitHub][3])

而且提示词明确要求：

* 选择**最相关**的指标
* 最多选 **8 个**
* 要避免冗余
* 先调用 `get_stock_data` 获取 CSV
* 再调用 `get_indicators`
* 写出“非常详细且细腻”的趋势报告
* 末尾加 Markdown 表格。([GitHub][3])

这说明这个节点不是死板地固定计算所有指标，而是让 LLM根据市场情境“挑指标”。

### 它的输出写到哪里

如果本轮已经不再有工具调用，结果文本会写入：

* `market_report`

并把消息对象返回到 `messages`。([GitHub][3])

### 它在系统中的位置

它负责回答的是：

* 价格趋势怎样
* 动量强不强
* 波动率如何
* 有没有技术性买卖点
* 当前走势更偏多头还是空头。([GitHub][3])

---

## 3. `news_analyst.py`

这个文件的作用是：**创建“新闻/宏观分析员”节点**。它定义 `create_news_analyst(llm)`，返回 `news_analyst_node(state)`。这个节点也从状态读取交易日期和标的上下文。([GitHub][4])

### 它能调用哪些工具

它绑定两个工具：

* `get_news`
* `get_global_news` ([GitHub][4])

和社交媒体 analyst 不同，这里明显把重点放在：

* 公司相关新闻
* 更广泛的宏观、全球市场新闻。([GitHub][4])

### 它的提示词在要求什么

`system_message` 里明确写着，它是一个 “news researcher”，任务是：

* 分析最近一周的新闻与趋势
* 写出一个与交易和宏观经济相关的“当前世界状态”综合报告
* 用 `get_news(query, start_date, end_date)` 做公司/主题新闻检索
* 用 `get_global_news(curr_date, look_back_days, limit)` 做宏观新闻检索
* 给出带证据的、可执行的洞见
* 在末尾附 Markdown 表格。([GitHub][4])

所以它关注的不是公司财报，也不是技术图形，而是“事件驱动”和“宏观环境”。

### 它的输出写到哪里

当工具调用完成后，它把最终文本写入：

* `news_report`

同时返回 `messages: [result]`。([GitHub][4])

### 它在系统中的位置

它主要回答：

* 最近有哪些公司相关新闻
* 宏观政策、利率、经济、全球事件是否影响这只股票
* 当前外部环境对交易偏多还是偏空。([GitHub][4])

---

## 4. `social_media_analyst.py`

这个文件的作用是：**创建“社交媒体/情绪分析员”节点**。它定义 `create_social_media_analyst(llm)`，返回 `social_media_analyst_node(state)`。同样，它读取 `trade_date` 和公司上下文。([GitHub][5])

### 它能调用哪些工具

它只绑定了一个工具：

* `get_news` ([GitHub][5])

这点很值得注意：虽然文件名叫 `social_media_analyst.py`，但从代码看，它并没有绑定一个单独的“社交媒体 API 工具”，而是复用了 `get_news`。也就是说，这个角色在概念上是“社媒与情绪分析员”，但底层数据入口目前仍是新闻/相关讨论搜索接口。这个判断直接来自源码中的工具列表。([GitHub][5])

### 它的提示词在要求什么

它的 `system_message` 要求模型：

* 分析某家公司过去一周的社交媒体帖子、公司相关新闻和公众情绪
* 写成长篇综合报告
* 关注公众每天对公司的看法和情绪变化
* 结合社媒、情绪和新闻来源
* 用 `get_news(query, start_date, end_date)` 去搜公司相关新闻和社媒讨论
* 给出对交易者有帮助的可执行洞见
* 末尾附 Markdown 表格。([GitHub][5])

### 它的输出写到哪里

如果这轮不再需要工具，结果文本会写入：

* `sentiment_report`

并把消息对象返回到 `messages`。([GitHub][5])

### 它在系统中的位置

它负责回答：

* 大众情绪偏多还是偏空
* 市场叙事热度如何
* 是否有社媒炒作、争议、恐慌或乐观情绪
* 舆情是否会影响短线交易。([GitHub][5])

## 5. 这四个文件的共同结构

这 4 个文件的代码框架几乎是同一个模板，只是工具和 prompt 不同。共同点包括：

1. 都定义 `create_xxx_analyst(llm)` 工厂函数。([GitHub][2])
2. 工厂函数都返回一个 `xxx_analyst_node(state)`，供 graph 注册为节点。([GitHub][2])
3. 都从 `state` 中读取：

   * `trade_date`
   * `company_of_interest`。([GitHub][2])
4. 都调用 `build_instrument_context(...)` 生成标的上下文。([GitHub][2])
5. 都使用 `ChatPromptTemplate.from_messages(...)` + `MessagesPlaceholder(variable_name="messages")`，说明它们都是在共享对话上下文上继续推理。([GitHub][2])
6. 都通过 `llm.bind_tools(tools)` 让模型具备 ReAct/tool-calling 能力。([GitHub][2])
7. 都有相同的协作型基础 system prompt，强调：

   * 和其他 assistant 协作
   * 用工具推进任务
   * 如果还不能完整回答没关系，后续 agent 会继续
   * 如果形成最终交易提案，要用 `FINAL TRANSACTION PROPOSAL: **BUY/HOLD/SELL**` 前缀。([GitHub][2])

所以这四个文件不是四套完全不同的架构，而是**同一 analyst 节点模板在不同分析视角上的实例化**。

---

## 6. 四个文件之间的分工差异

最简化地说：

* `market_analyst.py`：看**价格与技术指标**，输出 `market_report`。([GitHub][3])
* `social_media_analyst.py`：看**情绪与舆情**，输出 `sentiment_report`。([GitHub][5])
* `news_analyst.py`：看**公司新闻与宏观新闻**，输出 `news_report`。([GitHub][4])
* `fundamentals_analyst.py`：看**基本面与财务报表**，输出 `fundamentals_report`。([GitHub][2])

这四份 report 后续会被 researcher、trader、portfolio manager 等更高层节点继续使用，这一点和你前面分析 `graph` 目录时看到的整体流程是一致的：这些 analyst 是图前半段的信息采集和初步分析层。([GitHub][2])

---

## 7. 一个你读源码时会注意到的小点

这四个文件顶部都导入了：

* `time`
* `json`
* `get_config`

但从当前可见源码里，这几个名字在函数主体中并没有实际使用。也就是说，**这些 import 看起来更像历史遗留或为后续扩展预留，而不是当前核心逻辑的一部分。** 这个判断来自当前文件内容本身。([GitHub][2])

## 一句话总结

这个文件夹里的 4 个文件，作用就是：**分别定义市场、情绪、新闻、基本面四类 analyst 节点，让它们各自调用专属工具，生成四份不同维度的分析报告，供后续 researcher 和 trader 节点继续综合判断。** ([GitHub][2])

可以。这个 `tradingagents/agents/utils` 文件夹，作用是给整套多智能体系统提供 **共享状态定义、通用辅助函数、工具封装和记忆模块**。目录里目前有 8 个文件：

* `agent_states.py`
* `agent_utils.py`
* `core_stock_tools.py`
* `fundamental_data_tools.py`
* `memory.py`
* `news_data_tools.py`
* `technical_indicators_tools.py`
* `各文件的作用.md`。([GitHub][1])

你可以把它理解成：前面那些 analyst、researcher、manager、risk debator 文件负责“角色行为”，而这个 `utils` 目录负责“它们共同依赖的底层支撑”。

---

## 1. `agent_states.py`

这个文件的作用是：**定义整张 LangGraph 共享状态的数据结构**。它在里面定义了 3 个核心类型：

* `InvestDebateState`
* `RiskDebateState`
* `AgentState`。([GitHub][2])

### `InvestDebateState` 做什么

它描述“是否投资”的多空辩论状态，包含：

* `bull_history`
* `bear_history`
* `history`
* `current_response`
* `judge_decision`
* `count`。([GitHub][2])

也就是说，bull researcher 和 bear researcher 来回辩论时，相关历史和当前回应都存这里。

### `RiskDebateState` 做什么

它描述“风险管理辩论”的状态，包含：

* `aggressive_history`
* `conservative_history`
* `neutral_history`
* `history`
* `latest_speaker`
* `current_aggressive_response`
* `current_conservative_response`
* `current_neutral_response`
* `judge_decision`
* `count`。([GitHub][2])

这正好对应你前面看过的 `aggressive_debator.py`、`conservative_debator.py`、`neutral_debator.py` 三方轮转讨论。

### `AgentState` 做什么

`AgentState` 继承 `MessagesState`，相当于整个交易图的总状态容器。它除了消息列表外，还定义了全流程共享字段，比如：

* `company_of_interest`
* `trade_date`
* `sender`
* `market_report`
* `sentiment_report`
* `news_report`
* `fundamentals_report`
* `investment_debate_state`
* `investment_plan`
* `trader_investment_plan`
* `risk_debate_state`
* `final_trade_decision`。([GitHub][2])

所以一句话说，`agent_states.py` 是整套 MAS 的 **状态 schema 文件**。没有它，graph 里各节点就没有统一的数据接口可读写。([GitHub][2])

## 2. `agent_utils.py`

这个文件的作用是：**提供 agent 层共用的小工具函数，并统一导入常用工具对象**。从源码看，它做了两件核心事情：

* 导入工具函数：`get_stock_data`、`get_indicators`、`get_fundamentals`、`get_balance_sheet`、`get_cashflow`、`get_income_statement`、`get_news`、`get_insider_transactions`、`get_global_news`
* 定义 `build_instrument_context()` 和 `create_msg_delete()`。([GitHub][3])

### `build_instrument_context(ticker)` 做什么

它会返回一段简短说明，强调：

* 当前分析对象是哪个 ticker
* 在每次 tool call、report 和 recommendation 中都要使用**完全一致的 ticker**
* 还要保留交易所后缀，例如 `.TO`、`.L`、`.HK`、`.T`。([GitHub][3])

这个函数很重要，因为它能减少 LLM 把 `7203.T` 写成 `7203`、或者把带交易所后缀的股票误写成美股 ticker 的问题。

### `create_msg_delete()` 做什么

它返回一个内部函数 `delete_messages(state)`。这个函数会：

1. 读取 `state["messages"]`
2. 为每条消息生成 `RemoveMessage(id=...)`
3. 再追加一条占位的 `HumanMessage(content="Continue")`
4. 返回新的 messages 更新结果。([GitHub][3])

源码注释明确写着：这是为了 **清空消息，并加一个 Anthropic 兼容的最小占位消息**。([GitHub][3])

这和你前面在 `graph/setup.py` 里看到的 `Msg Clear Market`、`Msg Clear Social`、`Msg Clear News`、`Msg Clear Fundamentals` 节点正好对应上：这些清理节点本质上就是用这个函数生成的。

所以，`agent_utils.py` 本质上是 **agent 层的通用辅助模块**。([GitHub][3])

## 3. `core_stock_tools.py`

这个文件的作用是：**把“获取股票价格数据”封装成 LangChain 可调用工具**。它定义了一个带 `@tool` 装饰器的函数：

* `get_stock_data(symbol, start_date, end_date)`。([GitHub][4])

这个函数的内部逻辑很简单：不自己抓数据，而是调用：

```python
route_to_vendor("get_stock_data", symbol, start_date, end_date)
```

也就是把请求交给 `dataflows/interface.py` 去按配置选择具体 vendor。([GitHub][4])

所以它的定位不是“实现股价抓取逻辑”，而是：

* 把数据层能力包装成 agent 可以调用的工具
* 统一参数描述
* 让 analyst 节点能用 tool-calling 的方式取 OHLCV 数据。([GitHub][4])

你前面在 `market_analyst.py` 里看到的 `get_stock_data`，实际上就是从这里来的。

## 4. `fundamental_data_tools.py`

这个文件的作用是：**把基本面和财务报表相关的数据接口包装成 agent 可调用工具**。它定义了 4 个 `@tool` 函数：

* `get_fundamentals`
* `get_balance_sheet`
* `get_cashflow`
* `get_income_statement`。([GitHub][5])

它们都不自己实现抓数，而是统一调用 `route_to_vendor(...)`：

* `get_fundamentals` → `route_to_vendor("get_fundamentals", ticker, curr_date)`
* `get_balance_sheet` → `route_to_vendor("get_balance_sheet", ticker, freq, curr_date)`
* `get_cashflow` → `route_to_vendor("get_cashflow", ticker, freq, curr_date)`
* `get_income_statement` → `route_to_vendor("get_income_statement", ticker, freq, curr_date)`。([GitHub][5])

所以这个文件是 analyst 层和 dataflows 层之间的一个“工具胶水层”：

* 上层 fundamentals analyst 只需要知道有这些工具
* 底层究竟走 yfinance 还是 Alpha Vantage，由 `route_to_vendor` 和配置决定。([GitHub][5])

一句话说，它是 **基本面工具封装层**。

## 5. `memory.py`

这个文件的作用是：**提供一个离线、基于 BM25 的金融情境记忆系统**。这个文件很关键，因为前面很多 researcher、manager、trader 节点都会传入 `memory` 对象，并调用 `get_memories(...)`。源码开头就说明了它的设计目标：

* 使用 BM25 做词法相似度匹配
* 不依赖 API 调用
* 没有 token 限制
* 可离线工作
* 可配合任何 LLM provider。([GitHub][6])

### 核心类：`FinancialSituationMemory`

这个类内部维护：

* `self.documents`
* `self.recommendations`
* `self.bm25`。([GitHub][6])

### `_tokenize(text)` 做什么

它用正则 `\b\w+\b` 对文本做简单分词并转小写。也就是说，这个记忆系统不是 embedding 向量检索，而是传统 BM25 词项匹配。([GitHub][6])

### `_rebuild_index()` 做什么

每次新增 situation 后，它会对所有文档重新分词，并重建 `BM25Okapi` 索引。([GitHub][6])

### `add_situations(...)` 做什么

它接收一组 `(situation, recommendation)` 二元组，把它们分别加入 `documents` 和 `recommendations`，然后重建索引。([GitHub][6])

### `get_memories(current_situation, n_matches=1)` 做什么

这是最核心的方法。它会：

1. 对当前情境分词
2. 用 BM25 计算与所有历史情境的匹配得分
3. 取 top-n
4. 返回包含这些字段的结果：

   * `matched_situation`
   * `recommendation`
   * `similarity_score`。([GitHub][6])

分数还会做归一化处理，范围趋近于 0 到 1。([GitHub][6])

### `clear()` 做什么

清空所有记忆与索引。([GitHub][6])

所以 `memory.py` 的定位非常明确：它给系统提供了一种 **可持续积累“过去类似市场情境 → 当时经验建议”映射关系** 的轻量级记忆机制。([GitHub][6])

这也是为什么前面你看到：

* bull researcher
* bear researcher
* research manager
* trader
* portfolio manager

都会先构造 `curr_situation`，再去 memory 里找相似经验。

## 6. `news_data_tools.py`

这个文件的作用是：**把新闻相关接口包装成 agent 可调用工具**。它定义了 3 个 `@tool` 函数：

* `get_news(ticker, start_date, end_date)`
* `get_global_news(curr_date, look_back_days=7, limit=5)`
* `get_insider_transactions(ticker)`。([GitHub][7])

每个函数的内部实现都很简单，统一走 `route_to_vendor(...)`：

* `get_news` → `route_to_vendor("get_news", ticker, start_date, end_date)`
* `get_global_news` → `route_to_vendor("get_global_news", curr_date, look_back_days, limit)`
* `get_insider_transactions` → `route_to_vendor("get_insider_transactions", ticker)`。([GitHub][7])

所以这个文件的定位和 `fundamental_data_tools.py` 一样，也是一个 **工具包装层**。不同的是，它服务的是：

* `news_analyst.py`
* `social_media_analyst.py`
* 以及任何需要新闻或内幕交易信息的角色。([GitHub][7])

## 7. `technical_indicators_tools.py`

这个文件的作用是：**把技术指标查询封装成 agent 可调用工具**。它定义了一个 `@tool` 函数：

* `get_indicators(symbol, indicator, curr_date, look_back_days=30)`。([GitHub][8])

### 它的特殊点

它不仅仅是单纯地转发参数。源码里有一段专门处理：

* 如果 LLM 把多个指标写成一个逗号分隔字符串，例如 `"rsi, macd, atr"`，
* 它会自动拆分成多个指标，
* 对每个指标分别调用 `route_to_vendor("get_indicators", ...)`，
* 最后把结果用 `"\n\n".join(results)` 拼回来。([GitHub][8])

如果只传单个指标，就直接转发。([GitHub][8])

这很实用，因为 LLM 很容易一次把多个指标塞给同一个参数，这个工具专门做了容错。

所以 `technical_indicators_tools.py` 的作用是：

* 给 market analyst 提供技术指标工具
* 同时在工具层面处理“多指标误传”为单字符串的情况。([GitHub][8])

## 8. 这个 utils 文件夹整体怎么分工

把这些文件合起来看，分工其实很清楚：

### 状态层

* `agent_states.py`：定义 graph 里共享状态对象和两个辩论状态子结构。([GitHub][2])

### 辅助函数层

* `agent_utils.py`：提供 ticker 上下文构造和消息清理函数。([GitHub][3])

### 工具封装层

* `core_stock_tools.py`：股票价格工具。([GitHub][4])
* `technical_indicators_tools.py`：技术指标工具。([GitHub][8])
* `fundamental_data_tools.py`：基本面和三大报表工具。([GitHub][5])
* `news_data_tools.py`：新闻、宏观新闻、内幕交易工具。([GitHub][7])

### 记忆层

* `memory.py`：BM25 记忆检索系统。([GitHub][6])

也就是说，这个目录并不直接承担“某个角色的推理逻辑”，而是承担 **所有角色共用的基础设施**。

---

## 9. 和你前面看过的目录怎么连起来

这个 `utils` 目录和前面你分析过的目录关系非常紧：

* `analysts` 文件夹里的 market/social/news/fundamentals analyst 会用这里的 tool 封装函数。([GitHub][4])
* `graph/setup.py` 里的 `Msg Clear Xxx` 节点，本质上依赖这里的 `create_msg_delete()`。([GitHub][3])
* `researchers`、`managers`、`trader` 都依赖这里的 `FinancialSituationMemory`。([GitHub][6])
* 整个 graph 的共享 `AgentState` 也由这里定义。([GitHub][2])

所以它是一个非常典型的“横向公共模块”。

## 一句话总结

这个 `tradingagents/agents/utils` 文件夹里的各文件作用分别是：

* `agent_states.py`：定义整个 MAS 的共享状态结构。([GitHub][2])
* `agent_utils.py`：提供 ticker 上下文构造和消息清理工具。([GitHub][3])
* `core_stock_tools.py`：封装股价数据工具。([GitHub][4])
* `fundamental_data_tools.py`：封装基本面和报表工具。([GitHub][5])
* `memory.py`：提供 BM25 离线记忆检索。([GitHub][6])
* `news_data_tools.py`：封装新闻、宏观新闻、内幕交易工具。([GitHub][7])
* `technical_indicators_tools.py`：封装技术指标工具，并支持多指标字符串拆分。([GitHub][8])
* `各文件的作用.md`：当前为空。([GitHub][9])

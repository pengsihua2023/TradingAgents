这个文件夹目前有 2 个 Python 文件：

* `research_manager.py`
* `portfolio_manager.py`

另外还有一个 `各文件的作用.md`，但当前这个 Markdown 文件是空的，没有实际内容。([GitHub][1])

你可以把这个文件夹理解成：**前面 analyst 和 debater 产生了大量分析与辩论结果之后，这里负责“裁决、整合、落地”**。其中：

* `research_manager.py` 负责对 **多空投资辩论** 做总结和裁决，形成给 trader 的投资计划。([GitHub][2])
* `portfolio_manager.py` 负责对 **风险偏好辩论** 做最终整合，输出整个系统最后的交易决策。([GitHub][3])

下面逐个说。

---

## 1. `research_manager.py`

这个文件的作用是：**创建“研究经理 / 投资辩论裁判”节点**。它定义了 `create_research_manager(llm, memory)`，返回内部函数 `research_manager_node(state)`，说明它是一个图节点工厂函数，供上层 graph 注册使用。([GitHub][2])

### 它从 state 里读取什么

这个节点会从共享状态中读取这些关键内容：

* `company_of_interest`
* `investment_debate_state`
* `market_report`
* `sentiment_report`
* `news_report`
* `fundamentals_report`。([GitHub][2])

也就是说，它不是直接抓数据，也不再调用工具，而是站在更高层，读取前面 analyst 已经生成好的 4 份报告，以及 bull/bear 的辩论历史。([GitHub][2])

### 它做了什么额外处理

它先调用 `build_instrument_context(state["company_of_interest"])` 生成标的上下文；然后把四份报告拼成一个 `curr_situation` 文本，再用 `memory.get_memories(curr_situation, n_matches=2)` 取出两条最相关的历史经验，把这些历史反思拼成 `past_memory_str`。这说明这个节点不是只看当前辩论，还会参考“过去类似情境下曾犯过什么错”。([GitHub][2])

### 它的 prompt 在要求什么

这个文件里的 prompt 非常明确：
它要求 LLM 以“portfolio manager and debate facilitator”的身份，对这一轮辩论做**明确裁决**：

* 在 bull analyst 和 bear analyst 之间做出决定
* 只有在“强有力理由”下才允许选 Hold
* 简明总结双方最有力的论点
* 给出清晰可执行的建议：Buy、Sell 或 Hold
* 再额外生成一份 **detailed investment plan for the trader**
* 同时参考过去类似错误，改进当前判断。([GitHub][2])

也就是说，这个节点的核心任务不是“继续争论”，而是 **结束投资辩论，形成裁决和执行计划**。([GitHub][2])

### 它输出什么

它调用 `llm.invoke(prompt)` 得到结果后，会更新 `investment_debate_state`，其中至少包括：

* `judge_decision`
* `history`
* `bear_history`
* `bull_history`
* `current_response`
* `count`。([GitHub][2])

同时，它把 `response.content` 写到：

* `investment_plan`。([GitHub][2])

所以这个文件的最终产物可以理解为两部分：

1. **更新后的投资辩论状态**
2. **给 trader 的投资计划文本**。([GitHub][2])

### 它在整套系统里的位置

它相当于：

* bull researcher 和 bear researcher 的“裁判”
* 从四份基础报告和多空辩论中提炼结论的人
* trader 的上游决策提供者。([GitHub][2])

一句话说，`research_manager.py` 负责把“研究和辩论”变成“投资计划”。([GitHub][2])

---

## 2. `portfolio_manager.py`

这个文件的作用是：**创建“组合经理 / 最终交易决策者”节点**。它定义了 `create_portfolio_manager(llm, memory)`，返回内部函数 `portfolio_manager_node(state)`。和 `research_manager.py` 一样，它也是一个供上层图注册的 manager 节点工厂。([GitHub][3])

### 它从 state 里读取什么

这个节点会读取：

* `company_of_interest`
* `risk_debate_state`
* `market_report`
* `news_report`
* `fundamentals_report`
* `sentiment_report`
* `investment_plan`。([GitHub][3])

这说明它的输入层级比 `research_manager.py` 更靠后：
它不只接收 analyst 报告，还接收前面 `research_manager` 已经形成的 `investment_plan`，再结合风险分析辩论来做最后判断。([GitHub][3])

### 它也会调取记忆

和 `research_manager.py` 一样，它先把四份报告拼成 `curr_situation`，然后用 `memory.get_memories(curr_situation, n_matches=2)` 检索两条历史经验，再组装成 `past_memory_str`。这说明最终交易决策也会参考过去类似场景下的经验教训。([GitHub][3])

### 它的 prompt 在要求什么

这个文件里的 prompt 比 `research_manager.py` 更接近最终输出模板。它明确要求模型以 “Portfolio Manager” 身份：

* 综合风险分析员们的辩论
* 输出**最终交易决策**
* 使用固定的五档评级之一：

  * Buy
  * Overweight
  * Hold
  * Underweight
  * Sell
* 同时给出：

  1. `Rating`
  2. `Executive Summary`
  3. `Investment Thesis`。([GitHub][3])

它还特别要求结合：

* `Trader's proposed plan`
* `Lessons from past decisions`
* `Risk Analysts Debate History`。([GitHub][3])

所以这个节点不是简单复述 trader 计划，而是：**在 trader 计划基础上，再经过风险层的激进/保守/中立讨论后，给出最终的组合级决策和仓位倾向**。([GitHub][3])

### 它输出什么

执行后，它会构造新的 `risk_debate_state`，保留并更新这些字段：

* `judge_decision`
* `history`
* `aggressive_history`
* `conservative_history`
* `neutral_history`
* `latest_speaker`
* `current_aggressive_response`
* `current_conservative_response`
* `current_neutral_response`
* `count`。([GitHub][3])

然后把最终文本写到：

* `final_trade_decision`。([GitHub][3])

所以它的核心产物就是系统最后的交易决策文本。([GitHub][3])

### 它在整套系统里的位置

它相当于：

* 风险辩论层的终审者
* trader 计划的最后把关人
* 整个 MAS 最终给出 Buy/Hold/Sell 等评级的人。([GitHub][3])

一句话说，`portfolio_manager.py` 负责把“投资计划 + 风险辩论”变成“最终交易决策”。([GitHub][3])

---

## 3. 这两个文件的分工差异

最简单地说：

### `research_manager.py`

负责 **投资研究层面的裁决**：

* 总结 bull / bear 辩论
* 明确站队
* 给出 Buy / Sell / Hold 建议
* 形成 `investment_plan` 给 trader。([GitHub][2])

### `portfolio_manager.py`

负责 **组合与风险层面的最终裁决**：

* 综合风险分析员的讨论
* 参考 trader 的计划
* 给出最终五档评级
* 写出 `final_trade_decision`。([GitHub][3])

所以两者不是重复关系，而是上下游关系：

**Research Manager → Trader → Risk Debate → Portfolio Manager**。这个上下游关系，也和你前面分析 `graph/setup.py` 时看到的整体流程一致：研究经理先产出投资计划，组合经理最后收口输出终局决策。([GitHub][3])

---

## 4. 这两个文件的共同结构

它们两者的代码模式也非常像，都有这些共同点：

* 都是 `create_xxx_manager(llm, memory)` 工厂函数
* 都返回内部 `xxx_node(state)` 节点函数
* 都会调用 `build_instrument_context(...)`
* 都会从当前四份 analyst 报告拼接出 `curr_situation`
* 都会从 memory 中检索两条相似历史经验
* 都通过 `llm.invoke(prompt)` 直接生成自然语言决策结果
* 都把结果一部分写回 debate state，一部分写到专属输出字段。([GitHub][3])

这说明这个文件夹里的 manager 层，本质上是：
**带有记忆检索能力的高层裁决节点**。([GitHub][3])

---


## 一句话总结

这个 `managers` 文件夹里：

* `research_manager.py` 负责 **给投资辩论做裁决，并生成投资计划**。([GitHub][2])
* `portfolio_manager.py` 负责 **给风险辩论和 trader 计划做最终收口，输出最终交易决策**。([GitHub][3])

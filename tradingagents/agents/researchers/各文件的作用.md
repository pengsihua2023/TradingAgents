这个文件夹目前有 3 个文件：

* `bull_researcher.py`
* `bear_researcher.py`
* `各文件的作用.md`（当前为空文件） ([GitHub][1])

你可以把 `researchers` 这一层理解成：**在四类 analyst 产出基础报告之后，专门负责做“多空辩论”的研究员层**。前面的 analyst 更像“信息搜集与初步分析”，这里的 researcher 则是“拿着这些材料，分别站在看多和看空两边打辩论赛”。这两个文件都会读取 `market_report`、`sentiment_report`、`news_report`、`fundamentals_report`，再结合历史 memory，生成新的辩论内容，并更新 `investment_debate_state`。([GitHub][2])

---

## 1. `bull_researcher.py`

这个文件的作用是：**创建看多立场的研究员节点**。它定义了 `create_bull_researcher(llm, memory)`，返回内部函数 `bull_node(state)`。也就是说，它不是独立执行脚本，而是一个给上层 graph 注册的节点工厂函数。([GitHub][2])

### 它从 state 里读取什么

`bull_node(state)` 主要读取两类信息。

第一类是投资辩论状态 `investment_debate_state`，包括：

* `history`：整个投资辩论历史
* `bull_history`：bull 这边自己过去说过的话
* `current_response`：上一位发言者最近一次发言内容
* `count`：目前辩论轮数计数。([GitHub][2])

第二类是四份前置分析报告：

* `market_report`
* `sentiment_report`
* `news_report`
* `fundamentals_report`。([GitHub][2])

这说明 bull researcher 自己不再调用工具抓数据，而是建立在 analyst 层已经完成的研究成果之上。([GitHub][2])

### 它怎么使用 memory

代码会先把四份报告拼成一个 `curr_situation`，再调用：

```python
past_memories = memory.get_memories(curr_situation, n_matches=2)
```

然后把取回来的每条 `recommendation` 拼成 `past_memory_str`。这意味着 bull researcher 不只是现场发挥，还会参考过去相似情境下的经验和教训。([GitHub][2])

### 它的 prompt 在要求什么

这个文件的 prompt 非常明确：模型要扮演 “Bull Analyst advocating for investing in the stock”，也就是坚定站在看多一边。它需要重点强调：

* 成长潜力
* 竞争优势
* 正面市场指标
* 对 bear 观点的反驳
* 用对话式、辩论式风格与对方交锋，而不是只罗列事实。([GitHub][2])

prompt 还明确要求模型使用这些资源：

* 市场研究报告
* 社交媒体情绪报告
* 最新世界新闻
* 公司基本面报告
* 辩论历史
* 上一次 bear 的发言
* 相似历史情境中的反思与经验。([GitHub][2])

最后还特别写了一句：
**“You must also address reflections and learn from lessons and mistakes you made in the past.”**
也就是说，这个角色不仅要辩论，还要体现“从历史错误中学习”的能力。([GitHub][2])

### 它输出什么

LLM 返回结果后，代码会包装成：

```python
argument = f"Bull Analyst: {response.content}"
```

然后更新 `investment_debate_state`：

* `history`：追加本次 bull 发言
* `bull_history`：追加本次 bull 发言
* `bear_history`：保留原值
* `current_response`：设为当前 bull 发言
* `count`：在原有基础上加 1。([GitHub][2])

最后返回：

```python
{"investment_debate_state": new_investment_debate_state}
```

所以这个文件的本质作用就是：
**生成一段新的看多辩论内容，并把它写回投资辩论状态。** ([GitHub][2])

### 它在系统里的位置

它位于 analyst 层之后、research manager 之前。它不是最终拍板的人，而是多空辩论中的“多头辩手”。从整个系统流程看，它的作用是帮助后面的 `Research Manager` 看清楚最强的 bull case 是什么。这个定位也和 graph 层里 bull/bear 来回辩论后再进入 `Research Manager` 的结构一致。([GitHub][2])

## 2. `bear_researcher.py`

这个文件的作用是：**创建看空立场的研究员节点**。它定义了 `create_bear_researcher(llm, memory)`，返回内部函数 `bear_node(state)`。结构上与 `bull_researcher.py` 几乎完全对称。([GitHub][3])

### 它从 state 里读取什么

和 bull researcher 一样，它先读取 `investment_debate_state`，包括：

* `history`
* `bear_history`
* `current_response`
* `count`。([GitHub][3])

然后读取同样四份报告：

* `market_report`
* `sentiment_report`
* `news_report`
* `fundamentals_report`。([GitHub][3])

所以它和 bull researcher 的输入来源是一致的，只是解释方向相反。([GitHub][3])

### 它怎么使用 memory

它同样构造 `curr_situation`，再通过：

```python
memory.get_memories(curr_situation, n_matches=2)
```

取出两条相似情境经验，并把它们拼成 `past_memory_str`。这说明 bear researcher 也具备“从历史失败中吸取教训”的机制，而不是只做一次性静态判断。([GitHub][3])

### 它的 prompt 在要求什么

这个 prompt 要模型扮演 “Bear Analyst making the case against investing in the stock”，即站在看空一边，重点强调：

* 风险与挑战
* 竞争弱点
* 负面指标
* 对 bull 观点的批判性反驳
* 以辩论风格直接回应 bull，而不是简单列事实。([GitHub][3])

prompt 里还列出了它应该重点关注的风险来源，例如：

* 市场饱和
* 财务不稳定
* 宏观经济威胁
* 市场定位弱化
* 创新下滑
* 竞争对手威胁。([GitHub][3])

和 bull researcher 一样，它也被要求使用：

* 四份 analyst 报告
* 辩论历史
* 上一次 bull 发言
* 过去相似情境中的经验与反思。([GitHub][3])

### 它输出什么

执行后，它会把回复包装成：

```python
argument = f"Bear Analyst: {response.content}"
```

然后更新 `investment_debate_state`：

* `history`：追加本次 bear 发言
* `bear_history`：追加本次 bear 发言
* `bull_history`：保留原值
* `current_response`：设为当前 bear 发言
* `count`：加 1。([GitHub][3])

最后同样返回：

```python
{"investment_debate_state": new_investment_debate_state}
```

所以它的作用就是：
**生成一段新的看空辩论内容，并把它写回投资辩论状态。** ([GitHub][3])

### 它在系统里的位置

它是 bull researcher 的直接对手，负责把最强的 bear case 提出来，让系统不要只听多头叙事。它的存在让 `Research Manager` 在做投资裁决时，能同时看到“为什么该买”和“为什么不该买”的两套最强论证。([GitHub][3])

## 3. 这两个 researcher 文件的共同结构

这两个文件几乎是镜像关系，共同点很明显：

* 都是 `create_xxx_researcher(llm, memory)` 工厂函数。([GitHub][2])
* 都返回一个 `xxx_node(state)` 节点函数。([GitHub][2])
* 都读取相同的四份 analyst 报告。([GitHub][2])
* 都会读取和更新 `investment_debate_state`。([GitHub][2])
* 都会通过 memory 检索两条相似历史经验。([GitHub][2])
* 都直接用 `llm.invoke(prompt)` 生成自然语言辩论内容，而不是工具调用型 agent。([GitHub][2])
* 都会把输出前缀化成固定身份标签：

  * `Bull Analyst: ...`
  * `Bear Analyst: ...`。([GitHub][2])

所以这个文件夹整体上就是：
**多空辩论层的两个对抗式研究员节点。** ([GitHub][2])

---

## 4. 这层和 analyst / manager 的区别

如果跟你前面分析过的目录对比，分工会很清楚：

* `agents/analysts`：负责搜集并整理不同维度的事实材料，输出 4 份基础报告。
* `agents/researchers`：负责基于这些材料做多空对抗式论证。
* `agents/managers`：负责在辩论结束后做裁决、形成投资计划或最终交易决策。([GitHub][2])

也就是说，`researchers` 这一层更像法律庭审里的双方辩手，而不是法官，也不是调查员。

这两个文件顶部都导入了：

但从当前源码内容看，这几个导入并没有真正参与函数主体逻辑。说明这里可能存在历史遗留 import，或者作者曾经计划扩展日志/消息结构，后来没有继续使用。这个判断来自当前文件本身。([GitHub][2])

## 一句话总结

这个 `researchers` 文件夹里的核心作用是：

* `bull_researcher.py`：负责生成**最强的看多论证**。 ([GitHub][2])
* `bear_researcher.py`：负责生成**最强的看空论证**。 ([GitHub][3])
* 两者共同把 analyst 层的报告转化为一场有来有回的投资辩论，供后续 `Research Manager` 裁决。 ([GitHub][2])
